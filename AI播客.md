__2\. 核心架构逻辑：双流汇聚 \(Dual\-Pipeline Architecture\)__

本阶段的流程不再是单线的，而是\*\*“双流启动，最终汇聚”\*\*。

- __支流 1 \(灵魂流\)：__ 评论 \-> 分析师 \-> metadata\.json \(确立基调\)
- __支流 2 \(骨架流\)：__ PDF \+ 目录 \-> 切分器 \-> raw\_chapters \(确立内容\)
- __汇聚点：__ metadata\.json \+ raw\_chapters \-> 矿工 Agent \-> chapter\_xx\.json

__3\. Agent 建立列表 \(Agent Roster\)__

本阶段共包含 2 个 AI Agent。

__🤖 Agent 1: 评论分析师 \(The Tone Analyst\)__

- __职能：__ 舆情分析、人设侧写。
- __输入：__ 3 条读者评论。
- __输出：__ metadata\.json。
- __调用频率：__ 全书项目初始化时调用 __1 次__。

__⛏️ Agent 2: 章节内容矿工 \(The Chapter Miner\)__

- __职能：__ 深度阅读理解、逻辑拆解、金句挖掘。
- __输入：__ metadata\.json \(定调\) \+ 单章纯文本。
- __输出：__ chapter\_xx\.json。
- __调用频率：__ 每一章调用 __1 次__ \(N 章共 N 次\)。

__4\. 详细执行步骤 \(Step\-by\-Step Execution\)__

根据你的要求，我将“环节 1”拆解为两个并行的初始化动作。

__🟢 步骤 1：项目初始化 \(双轨并行\)__

此步骤目标是准备好“大脑（定调）”和“素材（文本块）”。

__1A\. 定调分析 \(Tone Analysis\)__

- __动作：__ 将你选好的 3 条评论填入 Prompt。
- __执行者：__ __Agent 1 \(评论分析师\)__。
- __逻辑：__
	1. 读取 3 条评论。
	2. 分析听众痛点和情绪爽点。
	3. 生成 metadata\.json 并存入 /output/config/。
	4. *关键产出：* 确定了 AI 播客是“犀利风”还是“治愈风”。

__1B\. 物理清洗与切分 \(Cleaning & Slicing\)__

- __动作：__ 解析 PDF 并按章节切割。
- __执行者：__ __Node\.js 代码__ \(你写的脚本\)。
- __逻辑：__
	1. __读取目录：__ 从 toc\.pdf 或你整理的配置中，获取每一章的 Start\_Page 和 End\_Page。
	2. __提取文本：__ 使用 pdf\-parse 提取指定页码范围的文字。
	3. __正则去噪：__ 去除页眉（如书名重复）、页脚（页码数字）、无效字符。
	4. __暂存：__ 将每章的纯文本保存为临时文件（如 /temp/raw\_ch01\.txt）。

__🟡 步骤 2：原子化提取 \(Atomic Extraction\)__

此步骤是本阶段的核心，AI 开始介入处理长文本。

- __动作：__ AI 带着“定调”去读“文本”。
- __执行者：__ __Agent 2 \(章节内容矿工\)__。
- __逻辑：__
	1. __读取输入：__ 加载 metadata\.json \(作为 System Prompt 上下文\) 和 /temp/raw\_ch01\.txt \(作为 User Input\)。
	2. __LLM 思考：__
		- *过滤：* 忽略与定调无关的废话。
		- *拆解：* 提取逻辑原子（现象\-机制\-解法）。
		- *摘录：* 锁定 3\-5 句原书金句。
	3. __生成：__ 输出结构化的 JSON 数据。

__🔴 步骤 3：结构化存储 \(Serialization\)__

此步骤是收尾工作，确保数据落地。

- __动作：__ 校验与存档。
- __执行者：__ __Node\.js 代码__。
- __逻辑：__
	1. __格式校验：__ 检查 Agent 输出的 JSON 是否合法。
	2. __文件写入：__ 将结果保存为 /output/processed/chapter\_01\.json。
	3. __循环控制：__ 如果还有下一章，返回步骤 2 继续处理；否则结束第一阶段。

__2\. 环节流程与顺序 \(Workflow Stages\)__

本阶段共包含 3 个核心环节，必须按顺序执行，形成“流水线”作业：

__环节 1：初稿生成 \(Drafting\)__

- __目标：__ 将冷冰冰的“逻辑原子”转化为有温度的“双人对话”。
- __执行者：__ __Agent 1 \(AI 编剧\)__。
- __核心动作：__ 依据“4A 疗愈模型”（觉察\-接纳\-分析\-行动）构建对话骨架，嵌入金句。

__环节 2：演出润色 \(Polishing/Directing\)__

- __目标：__ 将书面语转化为“口语”，注入“灵魂”。
- __执行者：__ __Agent 2 \(AI 导演\)__。
- __核心动作：__ 加入语气词（“嗯\.\.\.”、“那个”）、标注情绪（mood: sad）、控制节奏（打断/抢话）。

__环节 3：合规与质检 \(Reviewing\)__

- __目标：__ 守住安全红线，确保不无聊。
- __执行者：__ __Agent 3 \(AI 审核\)__。
- __核心动作：__ 检查自杀/医疗敏感词（安全），检查长难句（质量）。

__4\. Agent 建立列表 \(Agent Roster\)__

本阶段 AI Agent 的核心 Prompt 逻辑微调如下：

__✍️ Agent 1: 编剧 \(The Screenwriter\)__

- __输出调整：__ 不需要关注音色 ID，只需要专注写出 Alice 和 Dr\. Ye 的高质量对话。
- __Prompt 重点：__ 引用金句，遵循逻辑原子。

__🎬 Agent 2: 导演 \(The Performance Director\)__

- __输出调整：__ 你的任务是往文本里__加“佐料”__。
- __Prompt 重点：__

"请在文本中自然地加入口语填料，如‘那个\.\.\.’、‘嗯\.\.\.’、‘咳咳’。注意：不要使用 XML 标签（如 <break>），直接用标点符号（逗号、句号）和文字来控制节奏，因为豆包模型能自动理解文本语义。"

__⚖️ Agent 3: 审核 \(The Censor\)__

- __输入：__ 导演润色后的文本。
- __Prompt 重点：__ 检查敏感词。

__2\. 环节流程与顺序 \(Workflow Stages\)__

本阶段是纯工程实现，主要由 __Node\.js 代码逻辑__ 驱动，不涉及 LLM 的创造性生成。

__环节 1：协议适配与封装 \(Protocol Adaptation\)__

- __目标：__ 将通用的“剧本 JSON”转换为火山引擎专用的“API Payload”。
- __动作：__ 角色 ID 映射、参数注入（采样率、格式）、BGM 开关配置。

__环节 2：任务提交与生成 \(Submission & Generation\)__

- __目标：__ 调用云端算力生成音频。
- __动作：__ 发送 HTTPS POST 请求，获取 task\_id。

__环节 3：轮询与下载 \(Polling & Delivery\)__

- __目标：__ 获取最终文件。
- __动作：__ 循环检查任务状态，状态为 Success 时下载 MP3 到本地。

__3\. 涉及的 Agent/模块 \(Module Design\)__

在这个阶段，我们不需要“具备思考能力的 AI Agent”，而是需要两个“执行准确的代码模块（Worker）”。

__🔌 模块 1：载荷组装器 \(The Payload Builder\)__

- __职能：__ 翻译官。把“人看的剧本”翻译成“机器看的指令”。
- __逻辑：__
	- 读取 voice\_map\.json。
	- 遍历剧本，将 role: "Alice" 替换为 speaker: "zh\_female\_\.\.\."。
	- 组装 audio\_config 和 action: 3。

__📡 模块 2：API 调度员 \(The API Dispatcher\)__

- __职能：__ 通信兵。负责与火山引擎服务器握手。
- __逻辑：__
	- 处理鉴权（Token Header）。
	- 处理重试（Retry）机制（防止网络抖动）。
	- 处理文件流写入（Stream Write）。
