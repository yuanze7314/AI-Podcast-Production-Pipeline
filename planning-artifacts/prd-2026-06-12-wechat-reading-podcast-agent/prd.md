# PRD: 微信读书书籍 PDF 播客化 Agent

## 0. Document Purpose

本文档用于定义本地单机版 Web 工作台 MVP：将微信读书相关书籍 PDF、读者评论、个人笔记和角色/TTS 配置，转化为可审核、可编辑、可重跑的双人播客生产流程。本文面向产品规划、Agent 工作流设计、前后端开发和后续 BMAD Epics/Stories 拆解。技术实现细节集中记录在 `addendum.md`，本文聚焦用户价值、工作流、功能需求和验收标准。

## 1. Vision

长书籍内容结构复杂，直接让 AI 一次性生成播客脚本不稳定，容易出现章节遗漏、角色漂移、JSON 失效、内容质量不一致和 TTS 重跑成本过高等问题。本产品要把“书籍播客化”从一次性黑盒生成，变成一条可检查、可编辑、可重跑的 Agent 化生产流水线。

第一版产品是本地单机 Web 工作台。用户上传书籍 PDF，并补充读者评论、个人笔记、提示词、LLM API、豆包/火山 TTS 和角色声线配置后，系统自动完成目录识别、章节分析、评论洞察、播客策划、分轮次脚本生成、脚本审核、分章节音频生成和全集合成。人工审核重点放在章节脚本，章节切分和章节分析以报告验收为主。

如果 MVP 成功，它将成为平台方和知识内容生产者的播客内容生产基础设施：平台可以把书籍播客化作为增值能力，内容创作者可以将书籍、笔记和评论规模化转成短视频音频和完整播客。

## 2. Target User

### 2.1 Jobs To Be Done

- 平台运营者希望把书籍内容转成播客，扩展付费功能和内容消费场景。
- 知识内容生产者希望批量生产稳定质量的书籍讲解音频，减少从阅读到脚本到配音的手工成本。
- 内容审核者希望在关键节点看到结构化报告和脚本块，而不是面对不可解释的一次性生成结果。
- 创作者希望把高赞评论、真实读者困惑和个人笔记融入脚本，让播客不是单纯摘要，而是有真实讨论感。

### 2.2 Non-Users (v1)

- 不面向多人协同团队，MVP 不提供账号、权限、云端同步或多人审稿。
- 不面向移动端创作，MVP 优先桌面浏览器 Web 工作台。
- 不面向泛音频剪辑师，MVP 只覆盖书籍播客化生产，不做完整 DAW 或复杂音频后期。

### 2.3 Key User Journeys

- **UJ-1. 内容创作者把一本书转成可审核的播客项目。** 创作者在本地 Web 工作台创建项目，上传 PDF，补充读书笔记、高赞评论和 TTS 配置。系统识别目录并进入目录确认页。创作者调整章节名、合并或删除无效章节后确认，系统生成章节分析和评论洞察报告。价值落点是创作者获得一份可验收的内容结构，而不是等待黑盒音频。

- **UJ-2. 审核者编辑章节脚本并控制 TTS 成本。** 审核者打开某章脚本，看见按 Alice / Dr_Ye 对话轮次拆分的脚本块。审核者可修改文本、锁定满意轮次、重生成单轮或整章。所有脚本确认前不能进入 TTS。价值落点是 TTS 请求前已控制内容质量，减少失败重跑和无效成本。

- **UJ-3. 创作者导出短视频章节音频和完整播客。** 创作者确认所有章节脚本、开场、转场和结尾后生成分章节音频。系统记录每章 TTS 状态、失败原因和重跑入口，最后合成约 30 分钟完整播客。价值落点是同一项目同时产出短视频分发素材和完整播客音频。

## 3. Glossary

- **项目**：一本书的一次播客化生产任务，包含输入文件、配置、章节、分析报告、脚本、音频和任务日志。
- **书籍 PDF**：用户上传的书籍文件，是章节识别和内容提取的主要来源。
- **目录确认页**：系统识别章节后提供给用户验收和轻量编辑的页面。
- **章节分析报告**：每章的主题、核心观点、关键论证、可播客化问题和风险点。
- **评论洞察报告**：从高赞评论、读书笔记和划线中提取的情绪基调、真实案例、读者关注点和亮点。
- **播客策划**：综合章节分析、评论洞察和全书时长预算后的脚本创作蓝图。
- **脚本块**：按对话轮次拆分的最小可编辑单元，包含角色、文本、预计时长、TTS 声线和状态。
- **Alice**：主持人位，负责提问、推进、转场和代表用户困惑。
- **Dr_Ye**：同等地位的建议型朋友位，负责解释、类比、启发和建议。
- **Workflow**：确定性工程流程，负责 PDF 解析、TTS 封装、音频合成、状态机、重试和校验。
- **Agent**：需要判断、提炼、规划、创作或审核的 LLM 能力单元。

## 4. Features

### 4.1 本地项目管理与历史任务

**Description:** 用户可以在本地 Web 工作台创建、打开、继续和查看历史项目。每个项目保存输入、配置、中间产物、脚本状态、音频状态和错误日志。Realizes UJ-1, UJ-3.

#### FR-1: 创建本地播客项目

用户可以创建一个项目并上传书籍 PDF、评论/笔记文本、Agent 提示词配置、LLM API 配置、豆包/火山 TTS 配置和角色声线配置。

**Consequences (testable):**
- 创建项目后，系统生成唯一项目 ID 和本地项目目录。
- 项目配置可保存并在重新打开工作台后恢复。
- 缺少 LLM 或 TTS 配置时，系统标记对应步骤不可执行，并给出缺失项。

#### FR-2: 查看历史任务列表

用户可以查看历史项目列表，包括项目名称、书籍名、当前阶段、最近更新时间、已生成章节数、音频状态。

**Consequences (testable):**
- 用户关闭并重启服务后仍能看到历史项目。
- 用户能从历史列表进入项目详情并继续未完成流程。

### 4.2 PDF 摄取、章节识别与目录确认

**Description:** 系统从书籍 PDF 中提取文本，识别目录、章节标题、页码范围和正文片段，并提供目录确认页。用户可以修改章节名、合并章节、删除无效章节，然后确认进入分析。Realizes UJ-1.

#### FR-3: 解析 PDF 并生成候选章节

Workflow 必须从 PDF 中提取文本和候选章节结构，产出可审计的解析报告。

**Consequences (testable):**
- 解析结果包含章节标题、页码范围、文本长度、置信度和异常提示。
- 解析失败时保留错误日志，不影响用户重新上传或重跑。

#### FR-4: 目录确认页

用户可以在目录确认页验收章节结构，并进行轻量编辑。

**Consequences (testable):**
- 用户可以改章节名、合并章节、删除章节。
- 未确认目录前，章节分析步骤不可开始。
- 确认后的章节结构被持久化，并可作为后续 Agent 输入。

### 4.3 评论洞察与章节分析

**Description:** 系统分别处理读者评论/笔记和章节正文。评论洞察 Agent 提炼读者真实关注点、情绪基调、真实案例、亮点和争议点；章节分析 Agent 输出章节报告。用户主要验收报告，可对单章重新分析。Realizes UJ-1.

#### FR-5: 生成评论洞察报告

系统可以从用户提供的高赞评论、个人笔记和划线中提取播客可用洞察。

**Consequences (testable):**
- 报告包含情绪基调、读者困惑、真实案例、亮点句、争议点和可融入章节建议。
- 当评论输入为空时，系统仍可继续，仅标记评论洞察为空输入。

#### FR-6: 生成章节分析报告

系统可以为每章生成结构化分析报告。

**Consequences (testable):**
- 每章报告包含主题、核心观点、关键论证、可播客化问题、风险点。
- 用户可以重跑单章分析。
- 分析报告保留版本或最近一次重跑时间。[ASSUMPTION: MVP 只保留最近一次正式结果，完整版本历史进入 v2。]

### 4.4 播客策划与角色定调

**Description:** 播客策划 Agent 综合全书结构、章节分析、评论洞察和目标时长，生成整书播客蓝图。系统默认整书约 30 分钟，每章约 3-4 分钟；当章节数量过多时，按重要性压缩时长预算。Alice 和 Dr_Ye 的关系结构稳定，但具体语气随书籍类型动态设定。Realizes UJ-1, UJ-2.

#### FR-7: 生成整书播客策划

系统可以生成全书播客基调、章节时长预算、章节对话目标、案例/亮点融入计划、开场/转场/结尾策略。

**Consequences (testable):**
- 策划结果包含总时长预算和每章目标时长。
- 当章节过多导致总时长超出 30 分钟时，系统必须给出压缩策略。
- 策划结果引用评论洞察和章节分析中的具体要点。

#### FR-8: 动态生成 Alice / Dr_Ye 人设约束

系统根据书籍类型生成本项目的人设和口吻约束。

**Consequences (testable):**
- Alice 始终保持主持人位，Dr_Ye 始终保持建议型朋友位。
- 人设配置可被脚本生成和脚本审核复用。
- 不同书籍类型可以生成不同语气，但不能改变二者关系结构。

### 4.5 分轮次脚本生成、编辑与审核

**Description:** 编剧 Agent 根据播客策划生成每章双人对话脚本。脚本按轮次拆分，适配火山/豆包 TTS 的传输结构。用户可以逐轮编辑、锁定、重生成，并在确认后进入 TTS。脚本审核 Agent 检查角色漂移、JSON/schema、策划符合度、章节遗漏、自然度和时长。Realizes UJ-2.

#### FR-9: 生成分轮次脚本块

系统必须生成结构化脚本块，而不是单个大文本。

**Consequences (testable):**
- 每个脚本块至少包含角色、文本、预计时长、声线 ID、情绪/语速参数、状态。
- 脚本块可以序列化为稳定 JSON。
- 单章脚本目标时长默认 3-4 分钟，但必须服从整书时长预算。

#### FR-10: 脚本编辑与锁定

用户可以编辑、锁定、重生成脚本块。

**Consequences (testable):**
- 已锁定脚本块在重生成整章时不会被覆盖。
- 用户可以重生成单个脚本块或整章未锁定部分。
- 每章必须由用户确认后才能进入 TTS。

#### FR-11: 脚本审核

系统可以审核脚本并输出结构化问题列表。

**Consequences (testable):**
- 审核至少覆盖角色漂移、JSON/schema 合规、章节重点遗漏、对话自然度、时长超标。
- 审核失败时，系统标记问题位置并允许重跑相关脚本块。

### 4.6 开场、转场、结尾与音频生成

**Description:** 系统生成可编辑的整书开场、章节间转场和结尾总结，并在脚本确认后调用豆包/火山 TTS 生成分章节音频，最后合成完整播客。Realizes UJ-3.

#### FR-12: 生成可编辑串联内容

系统可以生成开场、章节间转场和结尾总结，并将其作为脚本块进入审核流程。

**Consequences (testable):**
- 开场、转场、结尾均可编辑和确认。
- 未确认串联内容时，全集合成不可开始。

#### FR-13: TTS 生成与失败重跑

系统在脚本确认后调用 TTS 生成分章节音频。

**Consequences (testable):**
- 未确认脚本不能发起 TTS。
- 每个 TTS 任务记录状态、失败原因、重跑入口和成本估算。
- 用户可以重跑失败章节，不必重跑整本书。

#### FR-14: 音频合成与导出

系统可以导出分章节音频和完整播客音频。

**Consequences (testable):**
- 每章音频可单独下载或用于短视频分发。
- 完整播客包含开场、章节音频、转场和结尾。
- 系统展示完整音频预计或实际总时长。

## 5. Non-Goals (Explicit)

- MVP 不做多人账号、权限、云端协作和团队审稿。
- MVP 不做移动端 App。
- MVP 不做复杂音频剪辑台、降噪、配乐混音或声音克隆训练。
- MVP 不承诺自动规避所有版权风险；版权合规和平台授权需作为业务侧前置条件。
- MVP 不做面向外部平台的开放 API，除非后续平台化阶段需要。

## 6. MVP Scope

### 6.1 In Scope

- 本地单机 Web 工作台。
- 项目管理和历史任务列表。
- PDF 上传、解析、目录确认。
- 评论/笔记洞察报告。
- 章节分析报告和单章重跑。
- 整书播客策划和时长预算。
- Alice / Dr_Ye 动态人设约束。
- 分轮次双人脚本生成、编辑、锁定、重生成和确认。
- 脚本审核和结构化问题提示。
- 可编辑开场、转场、结尾。
- 豆包/火山 TTS 生成分章节音频。
- 失败重跑、成本估算、全集音频合成。

### 6.2 Out of Scope for MVP

- 多用户协作和云端部署，延后到平台化版本。
- 完整版本历史和差异对比，除非脚本编辑阶段必须保留最小审计记录。
- 自动发布到播客平台或短视频平台。
- 精细音频后期制作。
- 企业级权限、审计、计费和 SLA。

## 7. Success Metrics

**Primary**

- **SM-1:** 端到端完成率：用户能从 PDF 创建项目并成功产出分章节音频和完整音频，目标 MVP 测试书籍完成率 >= 80%。Validates FR-1 through FR-14.
- **SM-2:** 脚本可用率：人工审核后，每章脚本需要整章重写的比例 <= 30%。Validates FR-7 through FR-11.
- **SM-3:** TTS 重跑成本控制：因脚本错误导致的 TTS 重跑章节比例 <= 20%。Validates FR-10, FR-11, FR-13.

**Secondary**

- **SM-4:** 章节结构验收效率：目录确认平均人工处理时间 <= 10 分钟/本书。[ASSUMPTION: 测试书籍为结构相对清晰的中文 PDF。] Validates FR-3, FR-4.
- **SM-5:** 输出时长控制：完整播客实际时长与目标 30 分钟偏差 <= 20%。Validates FR-7, FR-9, FR-14.

**Counter-metrics**

- **SM-C1:** 不以“全自动率”为第一优化目标。MVP 的目标是可控生产，不是减少所有人工介入。
- **SM-C2:** 不以“生成速度”牺牲脚本质量和 TTS 成本控制。

## 8. Open Questions

1. 火山/豆包 TTS 的最终请求 schema、角色声线字段、情绪/语速参数是否已有固定版本？
2. PDF 解析是否优先支持文本型 PDF，扫描版 OCR 是否进入 MVP？
3. 评论输入来自用户手动粘贴、文件导入，还是需要后续接入微信读书评论抓取？
4. 脚本块是否需要保留完整版本历史，还是 MVP 只保留当前版本与锁定状态？
5. 本地项目文件夹的可迁移性是否重要，例如复制到另一台机器能否继续打开？
6. 是否需要版权/合规提示页面，防止用户误以为系统自动解决授权问题？
